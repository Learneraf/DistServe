# 《面向异构集群的分离式推理框架》论文整体框架

本文档用于指导本科毕业论文写作。整体思路是：先在第 2 章讲清楚基础概念，再在第 3 章提出系统需求和问题建模，第 4 章说明系统总体设计，并在 4.5 节集中介绍异构资源搜索模块，第 5 章用实验结果说明方法是否有效以及为什么有效。

写作时要注意章节边界：

- 第 2 章讲背景概念：大语言模型推理、预填充、解码、KV 缓存、首词延迟、每词元生成时间、有效吞吐和 SLO。
- 第 3 章不再解释这些概念，只说明本文系统需要处理哪些对象、满足哪些约束，以及问题如何建模。
- 第 4 章写系统总体设计，并在 4.5 节集中介绍异构资源搜索模块和快速资源分配器。
- 不再单独设置“快速资源分配器设计”一章，也不单独设置“原型系统实现”一章。
- 第 5 章先给实验主结果，再用资源分配和请求流向解释结果来源。

本文建议统一使用以下表述：

- `benchmark` 写作“真实推理测试”或“实测数据”。
- `goodput` 首次写作“有效吞吐（goodput）”，后文写“有效吞吐”。
- `SLO` 首次写作“服务等级目标（SLO）”，后文写“延迟目标”或“SLO”。
- `handoff` 写作“KV 缓存传输”。
- `flow` 写作“请求流向”或“请求流量”。
- `shape` 写作“并行配置”。
- `allocator` 写作“资源分配器”。
- `fit` 写作“拟合集”，`val` 写作“验证集”。必要时可在括号里保留英文目录名。

## 论文题目

面向异构集群的分离式推理框架

## 摘要

### 中文摘要

第一段写研究背景。说明大语言模型在线推理需要同时满足低延迟和高吞吐，但预填充和解码两个阶段的计算特点不同，共置执行容易造成阶段间干扰。

第二段写本文问题。说明本文面向 CUDA GPU 与 Ascend NPU 组成的异构集群，研究如何在分离式推理框架下自动决定资源分配、并行配置和请求流向。

第三段写方法。概括本文基于实测数据建立执行时间模型，结合 KV 缓存传输开销和延迟目标，设计快速资源分配器来搜索异构部署方案。

第四段写结果。概括实验覆盖多个 LLaMA 模型和不同工作负载，并从有效吞吐、SLO 达标率、资源分配和请求流向几个角度验证方法效果。

### 英文摘要

对应中文摘要进行英文表达。重点术语建议统一为 disaggregated inference、heterogeneous cluster、prefill、decode、KV cache transfer、goodput、SLO attainment。

### 关键词

大语言模型推理；分离式推理；异构集群；资源调度；KV 缓存；有效吞吐

## 第 1 章 绪论

### 1.1 研究背景

介绍大语言模型在线推理服务的应用场景，例如智能问答、代码生成和内容生成。

说明在线推理和离线推理不同，在线服务不仅要处理更多请求，还要满足用户可感知的延迟要求。

引出预填充和解码两个阶段的差异，但这里只做背景铺垫，详细概念放到第 2 章。

### 1.2 研究意义

说明分离式推理的意义：把预填充和解码分开部署，可以减少阶段间干扰，并分别优化首词延迟和生成过程延迟。

说明异构集群的意义：真实环境中可能同时存在 CUDA GPU 和 Ascend NPU，不同硬件在不同阶段上的性能并不一致，固定使用一种设备或一种角色分工可能不是最优。

说明本文研究价值：让系统能够根据设备能力、延迟目标和 KV 缓存传输开销自动做资源分配，而不是依赖人工经验。

### 1.3 国内外研究现状

按方向综述相关工作：

1. 大语言模型推理服务系统，例如 Orca、vLLM。
2. 推理内存管理和连续批处理，例如 PagedAttention。
3. 分离式推理系统，例如 DistServe、Splitwise。
4. 异构集群调度和模型并行，例如 Pollux、Sia、Megatron-LM、Alpa。

最后指出本文切入点：已有分离式推理工作主要面向同构 GPU，异构调度工作又较少结合预填充/解码两阶段特点，因此需要面向异构集群重新建模和搜索。

### 1.4 本文主要工作

建议写成 3 到 4 点：

1. 设计面向 CUDA/Ascend 混合集群的分离式推理抽象，将服务实例划分为 CUDA 预填充、CUDA 解码、Ascend 预填充和 Ascend 解码四类。
2. 建立基于实测数据和小型推理仿真的服务能力建模流程，分别估计不同设备、不同角色和不同并行配置下的服务能力。
3. 设计快速资源分配器，在延迟目标和 KV 缓存传输约束下搜索实例数量和请求流向。
4. 在多个 LLaMA 模型和拟合集、验证集上进行实验，并用有效吞吐、SLO 达标率、资源分配和请求流向分析系统效果。

### 1.5 论文组织结构

简要介绍后续章节。

第 2 章介绍相关技术和基础概念；第 3 章给出系统需求和问题建模；第 4 章介绍系统总体设计和异构资源搜索模块；第 5 章进行实验评估；第 6 章总结全文并展望后续工作。

## 第 2 章 相关技术与基础

### 2.1 大语言模型推理流程

解释 Transformer 自回归推理的基本过程。

重点讲清楚预填充阶段和解码阶段：

- 预填充阶段一次处理完整输入提示词，生成后续解码需要的 KV 缓存。
- 解码阶段逐词元生成输出，并不断读取和更新 KV 缓存。

这一节要把概念讲完整，后续章节默认读者已经理解，不再重复解释。

### 2.2 推理服务性能指标

定义本文使用的指标：

- 首词延迟：请求发出到生成第一个词元的时间。
- 每词元生成时间：解码阶段平均生成一个词元的耗时。
- 请求总延迟：整个请求完成的时间。
- SLO 达标率：满足延迟目标的请求比例。
- 有效吞吐：在满足延迟目标前提下系统能承载的请求到达率。

说明为什么本文关注有效吞吐，而不是只关注普通吞吐。

### 2.3 vLLM 推理系统

介绍 vLLM 的作用和关键机制，包括连续批处理、PagedAttention、KV 缓存管理和服务接口。

说明本文使用 vLLM 获取 CUDA/Ascend 上的实测数据，并用于验证分离式推理在真实系统中的表现。

### 2.4 DistServe 分离式推理

介绍 DistServe 的核心思想：将预填充和解码部署到不同实例，分别满足首词延迟和每词元生成时间目标。

说明 DistServe 原始设计主要面向同构 GPU 集群，本文在此基础上加入 CUDA/Ascend 异构资源、跨设备 KV 缓存传输和请求流向选择。

### 2.5 异构集群与跨设备通信

介绍 CUDA GPU 和 Ascend NPU 在硬件能力、软件栈、内存管理和通信方式上的差异。

说明异构分离式推理中会出现跨设备池 KV 缓存传输，这部分开销会影响请求能否跨 CUDA 和 Ascend 执行。

本节只讲背景和问题来源，具体传输约束放到第 3 章建模。

### 3.2 工作负载与服务目标建模

```text
lambda = N / T
```

### 3.3 实测数据分析与服务能力建模

这一节单独对应 4.3 服务能力建模模块。写作重点是说明实测数据从哪里来、需要记录哪些字段，以及为什么只拟合公式还不够，必须放入小型推理仿真框架中评估。

说明实测数据来自 CUDA GPU 和 Ascend NPU 两类平台，需要覆盖不同模型、不同请求到达率，以及拟合集和验证集。

定义实测数据 `D_real`。说明它包括请求级延迟、预填充批处理数据和解码批处理数据。

请求级数据包括输入长度、输出长度、到达时间、首词延迟、每词元生成时间和总延迟。

预填充批处理数据包括批次大小、输入长度总和、最大输入长度、输入长度平方和，以及批次开始和结束时间。

解码批处理数据包括批次大小、上下文长度总和、最大上下文长度，以及解码轮次开始和结束时间。

说明需要实现一个小型推理仿真框架。框架维护预填充队列和解码队列，模拟请求到达、排队、预填充批处理、解码轮次推进、请求完成和 continuous batching。

说明服务能力建模模块不直接使用硬件峰值，而是根据实测数据拟合批次级执行时间：

```text
T_prefill = a_p + b_p * B + c_p * L_sum + d_p * L_max + e_p * L_sq
T_decode = a_d + b_d * B + c_d * C_sum + d_d * C_max
```

解释公式中特征的含义。预填充公式中的常数项表示固定开销，批次大小表示请求管理和调度开销，总输入长度表示主要计算量，最大输入长度表示批次对齐和最长序列影响，输入长度平方和用于刻画注意力计算的非线性增长。解码公式主要使用批次大小、上下文长度总和和最大上下文长度，因为解码阶段每轮基于已有 KV 缓存继续生成，主要受活跃请求数和上下文长度影响。

说明这里只给出需求层面的建模形式，具体参数拟合和连续批处理仿真放到 4.3 节展开。

定义单实例能力：

```text
mu_r = Capacity(r, s_r, W, T_ftl, T_tpot, D_real)
```

说明 `mu_r` 是服务能力建模模块给资源搜索模块的核心输出。

### 4.2 输入配置模块与实测数据采集模块

先说明输入配置模块负责统一模型、数据集、请求到达率、推理后端、设备平台和延迟目标。

说明本文使用 ShareGPT 请求作为工作负载，并针对 LLaMA 1B、LLaMA 3B、LLaMA 7B 和 LLaMA 8B 四个模型分别生成可运行请求。

说明数据集划分规则：从四个模型都能运行的公共请求集合中划分 fit、val 和 test。fit 用于拟合执行时间模型和服务能力参数，val 用于验证模型泛化效果，test 作为保留测试集合。每组包含 120 条请求，划分时按照输入长度和输出长度分桶，使不同集合的长度分布尽量接近。

说明请求到达率设置为 1 到 4 请求/秒，间隔 0.5；到达过程采用泊松到达，随机种子固定为 0。

说明 SLO 设置：先给出基准首词延迟目标和基准每词元生成时间目标，再使用 SLO 缩放系数生成不同严格程度的延迟目标。正文可以写使用 0.4、0.6、0.8、1.0 和 1.2 五个缩放系数。

说明本文使用 vLLM 和 DistServe 两个推理后端。vLLM 用于普通在线推理测试，体现连续批处理后端的运行情况；DistServe 用于预填充和解码分离部署测试，体现分离式推理框架的运行情况。CUDA GPU 和 Ascend NPU 是这两个后端运行的硬件平台。

再说明该模块负责在 CUDA 和 Ascend 上运行真实推理测试，得到不同模型、不同请求到达率和不同数据集下的延迟结果。

说明采集内容包括请求级结果、服务日志、批处理轨迹和必要的内部打点。

这一节只说明数据从哪里来、记录哪些信息，不展开脚本命令。

### 4.3 服务能力建模模块

说明该模块不是直接使用真实运行结果查表，而是用实测数据拟合批次执行时间，并基于 `simdistserve` 构建小型推理仿真框架，最终输出单实例服务能力。

解释为什么需要仿真：真实运行系统只能覆盖已经测试过的配置，资源搜索会产生更多候选方案；真实端到端延迟还混合了排队、调度和批处理影响，直接复用困难。

说明仿真框架不真实运行模型，而是使用拟合出的执行时间公式替代预填充和解码 forward 时间。

- 预填充批次时间与 batch size、总输入词元数、最大输入词元数、输入长度平方和有关。

#### 4.3.3 解码时间模型

说明解码轮次时间与 batch size、上下文长度总和、最大上下文长度等批次状态有关。

说明部分实验会加入剩余输出长度等扩展特征，但正文重点讲基础形式。

#### 4.3.4 参数拟合与验证

#### 4.5.1 设计目标

说明异构分离式推理需要同时决定三类内容：

1. 每类服务实例使用什么并行配置。
2. 每类服务实例部署多少个。
3. 请求在四类路径之间如何分配。

直接枚举所有组合开销较大。本文采用分层思路：先枚举并行配置并评估单实例能力，再用快速资源分配器求资源数量和请求流向。

#### 4.5.2 并行配置枚举

说明对四类角色分别枚举可行并行配置：

- CUDA 预填充
- CUDA 解码
- Ascend 预填充
- Ascend 解码

每个配置包含张量并行度、本地流水线并行度和跨节点流水线并行度。

合法性检查包括：实例占用设备数不能超过设备池容量，配置需要满足放置约束，模型在该配置下能够运行。

#### 4.5.3 单实例能力输入

说明资源分配器不直接模拟每个请求，而是使用上一章得到的单实例能力作为输入。

对于每类角色，需要知道一个实例在对应延迟目标下可承载的请求率。记为：

```text
mu_cp: CUDA 预填充单实例能力
mu_cd: CUDA 解码单实例能力
mu_ap: Ascend 预填充单实例能力
mu_ad: Ascend 解码单实例能力
```

#### 4.5.4 固定并行配置下的约束

定义实例数量：

```text
n_cp, n_cd, n_ap, n_ad
```

分别表示 CUDA 预填充、CUDA 解码、Ascend 预填充和 Ascend 解码实例数量。

写资源约束：

```text
CUDA 预填充占用 + CUDA 解码占用 <= CUDA 设备总数
Ascend 预填充占用 + Ascend 解码占用 <= Ascend 设备总数
```

写服务能力约束：

```text
x_cc + x_ca <= CUDA 预填充总能力
x_cc + x_ac <= CUDA 解码总能力
x_ac + x_aa <= Ascend 预填充总能力
x_ca + x_aa <= Ascend 解码总能力
```

写 KV 缓存传输约束：

```text
x_ca <= CUDA 到 Ascend 的传输上限
x_ac <= Ascend 到 CUDA 的传输上限
```

目标是最大化四条路径的总请求率。

#### 4.5.5 资源分配器核心方法

说明资源分配器分两步工作。

第一步，在每个设备池内部枚举“预填充实例数”和“解码实例数”的可行组合，并计算该设备池能提供的预填充总能力和解码总能力。只保留有意义的组合，例如资源占用更少但能力不低的组合。

第二步，把 CUDA 设备池组合和 Ascend 设备池组合配对。对于每个配对结果，把请求流向看成一个 2x2 网络，四条边分别对应 `x_cc`、`x_ca`、`x_ac` 和 `x_aa`。

在这个 2x2 网络里，有效吞吐受到四类上界限制：

```text
Lambda <= CUDA 预填充能力 + Ascend 预填充能力
Lambda <= CUDA 解码能力 + Ascend 解码能力
Lambda <= Ascend 预填充能力 + CUDA 解码能力 + CUDA->Ascend 传输上限
Lambda <= CUDA 预填充能力 + Ascend 解码能力 + Ascend->CUDA 传输上限
```

资源分配器取这些上界中的最小值作为该组合的最大可行有效吞吐，再恢复具体请求流向。最后选择有效吞吐最高的组合。如果多个组合相同，则优先选择资源使用更少或结构更简单的方案。

这一节要重点说明方法为什么快：它没有对四类实例数量和四类请求流量做大规模联合枚举，而是利用 2x2 请求流向结构快速计算每个组合的上界。

#### 4.5.6 基线方法

设置两个基线，便于说明本文方法的收益来自哪里。

##### 4.5.6.1 不跨设备池基线

只允许 `x_cc` 和 `x_aa`，禁止 CUDA 和 Ascend 之间传输 KV 缓存。

这个基线表示两个设备池各自独立服务，能说明跨设备池请求流向是否带来额外收益。

##### 4.5.6.2 固定异构分工基线

固定请求走 CUDA 预填充到 Ascend 解码，即只允许 `x_ca`。

这个基线表示一种人工指定的异构分工策略。它可以说明固定经验规则并不一定适合所有模型。

#### 4.5.7 算法复杂度分析

分析搜索开销主要来自三个部分：并行配置枚举、单实例能力评估、设备池内部组合枚举。

说明快速资源分配器降低的是实例数量和请求流向联合搜索的开销。

#### 4.5.8 输出结果解释

说明搜索结果包含：

- 四类实例数量。
- 四类请求流向。
- 估计有效吞吐。
- 单实例能力摘要。

这些结果会在第 5 章中用于画有效吞吐图、资源分配图和请求流向图。

1. 快速资源分配器可以根据模型和设备能力自动选择资源分配。

3. KV 缓存传输带宽会影响跨设备池执行的价值。
4. 单一固定策略无法覆盖所有模型。
